# Orca worktree adapter

Load this adapter only when the resolved artifact root has valid `.loom/config.json` with `{"worktrees":"orca"}`. OMP + Orca visible-TUI worker flow was verified live on 2026-07-21 through service commit `e81ef4db0d10bc574b46af59193200a59d770850` (tree `9662a2e8272ac822d52e32b5b79fd1ec931a1575`); other hosts remain unverified. The user starts one ordinary main OMP session at the project/workspace root.

## Native orchestration boundary

Load and follow the installed native Orca `orchestration` skill for task/DAG creation, dispatch/inject, blocking ask/reply, waits, retries, circuit breaker, heartbeat, liveness, and completion mechanics. This adapter adds only Loom payload, ordering, and Verify boundaries; do not duplicate or compete with native lifecycle logic.

Orca is the runner whenever this adapter is active. Goal is off: never offer or run OMP Goal simultaneously. Load and follow the canonical progressive [`STORY.md`](STORY.md) contract. A confirmed whole-pack preview activates native supervised orchestration only; APPROVE, issue completion, and whole-pack confirmation authorize no commit, push, hosted review/PR creation, merge, amend, squash, rebase, force, or other Git/host mutation. A new issue, repository, worktree action, or base renews confirmation. Plan creates no worktree or runtime task.

The user manually creates the top-level story coordinator worktree/card in Orca UI. Loom never creates it, exposes `/loom new`, or silently manufactures a STORY to make the current lane look like a coordinator. At every `/loom` entry, first resolve the validated workspace owner (`artifactRoot`) and compare it with Orca's native current repository/worktree/card context. The coordinator is the user-created top-level card whose worktree owns that validated artifact root. Any service worktree/card is a service lane even when it can see parent files. From a service lane, warn that `/loom` must run in the named coordinator card, return the native coordinator guidance, and make no STORY, task, lane, status, or Git mutation. Missing, duplicate, stale, or contradictory owner/context identity stops for human resolution.

## Story-service lanes

During adaptive Implement preview, identify touched registered repositories and preview every just-in-time worktree action with compact service scope, writer, native base, and current lane state. Create nothing until that repository addition/action is explicitly confirmed; classification or a whole-pack preview cannot silently confirm a later material repository change. Orca owns branch prefix, name, base, attribution, settings, paths, and repository/worktree/card/terminal IDs. Accept the actual branch and base returned by Orca; never impose `loom/`, persist checkout metadata in Loom files, or add a runtime/checkouts manifest. Worktree names, display text, and comments use product-facing story, service, and status language without Loom branding or private pack paths/IDs.

Create one story x service worktree just in time when that lane's first issue becomes runnable and its adaptive repository preview is confirmed, never during Plan, before confirmation, or merely because a pack was selected. Use the native Orca repository identity, default base, branch result, parent/lineage, attribution, and project settings unless the confirmed preview names a dependency base; never guess `main`. Rediscover lanes from Orca and Git product metadata. Zero matches permits the confirmed JIT create, one match resumes it, and multiple or inconsistent matches stop for human resolution. If Orca, registration, or config is unavailable, stop before Git changes with repair guidance.

## Scheduling and attempts

Task dependencies mirror each issue's `Blocked by` in Orca's native task DAG. One active writer owns each service lane. Within one repository, runnable issues and rework serialize by issue number unless explicit blockers require another order; no second writer dispatch starts until the prior assignment and independent Verify boundary finish. Service lanes in explicitly independent registered repositories may run in parallel. An explicitly atomic multi-repository issue acquires every listed lane and blocks conflicting work until its assignment and Verify finish. The root coordinator serializes all `.loom` Log, Verify, and status writes. Do not add a Loom scheduler, lane lock, or registry: native task/dispatch/liveness state is authoritative.

Keep one healthy visible OMP maker terminal per service lane and reuse it for bounded assignments. Lazy-load [`OMP.md`](OMP.md): create the lane's first terminal with `--prewalk --config <artifactRoot>/.omp/config.yml`; OMP 17.0.6 startup was live-verified to accept the flag/config and reach a ready visible TUI showing Prewalk, but no actual prewalk model switch was observed. Never enable Advisor in workers. The worker is maker only and must not invoke Verify/checkers or write root Loom artifacts. A completed assignment returns to an idle prompt; do not close the terminal after issue Verify. Replace it only when native Orca liveness says it is unhealthy or unavailable.

Every dispatch is one bounded issue or rework assignment. Initial dispatch contains the PRD plus that issue. Re-dispatch to an idle lane sends a compact delta containing the current task, confirmed decisions, current acceptance, authoritative diff/base since its last assignment, and relevant blocker/check changes. It sends neither the full private pack/transcript nor a task-only prompt. The completion payload names changed files/repositories, base SHA plus diff/tree identity, checks run, and concise decision/blocker log bullets.

A worker question uses native Orca ask/decision-gate mechanics. The coordinator bridges it to exactly one OMP AskUser question with a recommendation, then replies; only dependent tasks wait. If a load-bearing decision is unresolved, mark/route `needs-info`; never guess. The coordinator follows native orchestration lifecycle, waits, and liveness by reference rather than duplicating CLI mechanics. Loom adds only the Verify two-strikes rule and records final failure/escalation in the issue Log/status.

The coordinator accepts only `worker_done` corresponding to the active task/dispatch. That message ends one bounded assignment, not the issue, Verify, terminal, or lane. The coordinator rereads STORY, the PRD, issue, Git diff/tree, and Orca state, records maker evidence in `## Log`, and runs independent Loom Spec + Standards Verify. APPROVE alone may complete the issue and unblock dependents; it leaves STORY `open` and grants no Git/host mutation. REJECT keeps the same lane and re-dispatches its healthy idle maker with the compact delta; an unhealthy terminal is replaced through native Orca. A second REJECT with overlapping blockers stops the pack. Maker checks are evidence, never approval.

Use native card comments and status only at meaningful durable boundaries: confirmed decision or lane addition, assignment accepted/blocked, checks captured, Verify verdict, and explicit handoff. Do not comment for heartbeats, waits, background completion, or other live events. Background `worker_done` stays quiet until the coordinator records the durable result. After issue Verify, leave the healthy maker terminal idle and the card/worktree open; later explicit handoff/finish/Tend contracts own status transition and cleanup.

## Resume

The coordinator is disposable. Resume reads the validated current STORY, then obtains authoritative current `git status` and `git diff` independently in every registered service repository, then reads native Orca's story-filtered repository/worktree/card, task/dispatch, and terminal/liveness inventory. Transcripts are optional context only. Do not read them to establish state and do not create a resume manifest, lane registry, task registry, terminal registry, or orchestration engine.

Reconcile exact `(repository, repositoryId)` pair-set equality with one unique STORY lane, Git record, and native Orca lane for every exact registered repository record. Native `repositoryId` is authoritative identity; logical `repository` is only its validated label, so aliases, duplicate names/IDs, and pair mismatches stop. Each STORY/Orca lane must provide nonempty repository, native repositoryId, lane, task, terminal, card-status, and assignment identities; each Git record must provide repository, native repositoryId, status, diff summary, material-change list, and HEAD; unknown or missing fields are invalid. Compare the STORY and Orca native identities and compare Orca-observed HEAD with authoritative Git HEAD. Missing, duplicate, stale, unknown, or contradictory STORY/Git/Orca evidence is `STOP` before task creation or dispatch. State the exact mismatched source, repository, field, and values for user reconciliation; never guess through it. A coherent dirty uncommitted diff is normal resumable state, including after APPROVE: neither a commit nor a clean tree is required.

Only coherent evidence emits one compact actionable resume delta: current Goal; completed work; per-lane current Git status and diff summary; Open Questions; native lane/task/terminal/card inventory and current assignment; stale Verify; and material changes since the latest durable STORY boundary. Include the validated nonempty STORY next action. The current STORY owns durable decisions and boundary prose, Git owns current file state, and Orca owns live identities and mechanics; no source may overwrite another by inference. Reused lane makers and a fresh OMP handoff receive the same actionable fields, narrowed to their assignment.

Write a card comment/status only when the reconciled resume itself establishes a durable resume or explicit handoff boundary. Do not emit comments for scans, unchanged state, waits, compaction, or background activity.

## OMP context boundary

Before an expected native shake, persist only newly confirmed durable decisions or a newly completed cycle in the smallest owning artifact; do not rewrite unchanged state merely because compaction is expected. Native shake, host context-pressure, and observed decision-loss signals remain authoritative. Respect the root session's current Advisor setting, but never enable Advisor for makers or checkers. Makers use Prewalk. Goal remains off while Orca coordinates.

Native context pressure or observed decision loss returns one explicit handoff offer with no handoff. Confirmation is valid only after that offer and only with a validated nonempty actionable delta; it then uses native handoff mechanics. Missing, malformed, or confirmation-without-offer state stops. Loom never replaces the session automatically or repeats the offer without a new signal. On confirmation, use native terminal handoff mechanics to start a fresh OMP session in the same coordinator or service worktree and send the reconciled actionable delta since the last durable boundary. Record the explicit handoff as a durable card boundary. Without confirmation, keep the current session.

## Review reopen

Exact `review-feedback` is valid only from `awaiting-review`, requires explicit `changesRequired: true`, verbatim nonblank feedback, and a nonempty list of unique affected issues, and their repository lanes. It reuses the same story, lanes, worktrees, cards, and healthy terminals. Transition STORY to `open`, reopen only affected `done` issues whose latest verdict is APPROVE to `ready-for-agent`, mark only those Verify records stale, and return an append-only durable history proposal containing the exact feedback, affected IDs, and lifecycle transition for STORY/issue recording; append future work without rewriting existing commits, and record one durable reopen card boundary. Later finish work may append commits; this adapter does not implement those commits. Exact `service-merged` is valid only from `awaiting-review` with `changesRequired: false` and `feedback: null`; it proposes no history append; it changes neither lifecycle nor issue/Verify state. Other lifecycle/event combinations stop.

## Future story boundaries

Finish and publish are explicit future story-level authority boundaries. This adapter does not infer or implement them from APPROVE, issue completion, whole-pack confirmation, casual completion language, or card switching. Existing issue `done` and blocker behavior remains available without any commit or publication.

## Tend cleanup handoff

Explicit merged-worktree or local-lane cleanup routes to the existing Tend ritual. Tend first performs a read-only correlation of durable hosted-review merge evidence with exact Orca repository/worktree/card IDs, actual Git branch/HEAD and path, cleanliness, and active terminal/task/dispatch/rework state. A closed review, missing remote ref, issue status, or branch name is not merge proof. Missing, duplicate, orphaned, or contradictory disk/Git/Orca/host identity retains the lane for human resolution.

Only uniquely matched, durably merged, clean, inactive story-service lanes are removable by default, and only after a confirmation naming every local card/worktree/branch action. Tend removes that exact Orca-managed worktree only through native exact-selector `orca worktree rm`, then verifies the exact worktree/card is absent from both Orca and Git inventory. It retains closed-unmerged, dirty, blocked, active, rework, ambiguous, and orphaned lanes with an exact reason. If Orca cannot remove the worktree or either state cannot be verified, Tend retains the lane and branch and never falls back to `git worktree remove`, filesystem deletion, or another raw Git removal path. Only after verified native removal may Tend delete an exact still-present, still-eligible local branch with merged-safe deletion. It never deletes a remote branch by default, force-deletes a local branch, invokes `orca orchestration reset --all` or equivalent broad cleanup, or affects an unlisted terminal, task, worktree, or branch.

Cleanup is nontransactional per lane: preserve and report completed actions, identify partial failures, and rerun the exact read-only inventory before requesting confirmation for any retry. Internal consent chat may name exact local paths and IDs; product-facing summaries remain sanitized of private control-plane markers. Tend observes the human-performed merge and never crosses the human merge gate.
