# Orca worktree adapter

Load this adapter only when the resolved artifact root has valid `.loom/config.json` with `{"worktrees":"orca"}`. OMP + Orca visible-TUI worker flow was verified live on 2026-07-21 through service commit `e81ef4db0d10bc574b46af59193200a59d770850` (tree `9662a2e8272ac822d52e32b5b79fd1ec931a1575`); other hosts remain unverified. The user starts one ordinary main OMP session at the project/workspace root.

## Native orchestration boundary

Load and follow the installed native Orca `orchestration` skill for task/DAG creation, dispatch/inject, blocking ask/reply, waits, retries, circuit breaker, heartbeat, liveness, and completion mechanics. This adapter adds only Loom payload, ordering, Verify, and commit-tree boundaries; do not duplicate or compete with native lifecycle logic.

Orca is the runner whenever this adapter is active. Goal is off: never offer or run OMP Goal simultaneously. A confirmed whole-pack preview activates native supervised orchestration and is the single opt-in for exactly one post-APPROVE verified commit per issue; never request a second per-issue commit opt-in. Single-issue and other attended paths still require their applicable explicit commit opt-in. Pack confirmation never authorizes push, hosted review/PR creation, merge, amend, squash, rebase, or force. A new issue, repository, worktree action, or base renews confirmation. Declining pack commit authorization falls back to attended one-issue mode. Plan creates no worktree or runtime task.

## Story-service lanes

During Implement preview, identify touched registered repositories and preview every just-in-time worktree action. Orca owns branch prefix, name, base, attribution, settings, paths, and repository/terminal IDs. Accept the actual branch returned by Orca; never impose `loom/`, persist checkout metadata in Loom files, or add a runtime/checkouts manifest. Worktree names, display text, and comments use product-facing story, service, and status language without Loom branding or private pack paths/IDs.

Create one story x service worktree just in time when that lane's first issue becomes runnable, never during Plan or merely from preview. Use Orca's default base unless the confirmed plan names a dependency base; never guess `main`. Rediscover lanes from Orca and Git product metadata. Zero matches permits the confirmed JIT create, one match resumes it, and multiple or inconsistent matches stop for human resolution. If Orca, registration, or config is unavailable, stop before Git changes with repair guidance.

## Scheduling and attempts

Task dependencies mirror each issue's `Blocked by`. Within one service lane, runnable issues serialize by issue number unless explicit blockers require another order. Independent service lanes may run in parallel. An explicitly atomic multi-repository issue acquires every listed lane and blocks conflicting work until its attempt and Verify finish. The root coordinator serializes all `.loom` Log, Verify, and status writes.

Use a fresh visible OMP worker per issue, never reuse one. Lazy-load [`OMP.md`](OMP.md): launch every visible worker with `--prewalk --config <artifactRoot>/.omp/config.yml`; OMP 17.0.6 startup was live-verified to accept the flag/config and reach a ready visible TUI showing Prewalk, but no actual prewalk model switch was observed. Never enable Advisor in workers. The worker is maker only and must not invoke Verify/checkers or write root Loom artifacts. Its completion payload must name changed files/repositories, base SHA plus diff/tree identity, checks run, and concise decision/blocker log bullets.

A worker question uses native Orca ask/decision-gate mechanics. The coordinator bridges it to exactly one OMP AskUser question with a recommendation, then replies; only dependent tasks wait. If a load-bearing decision is unresolved, mark/route `needs-info`; never guess. The coordinator follows native orchestration lifecycle, waits, and liveness by reference rather than duplicating CLI mechanics. Loom adds only the Verify two-strikes rule and records final failure/escalation in the issue Log/status.

The coordinator accepts only `worker_done` corresponding to the active task/dispatch. That message ends one attempt, not the issue. The coordinator rereads the PRD, issue, Git diff/tree, and Orca state, records maker evidence in `## Log`, and runs independent Loom Spec + Standards Verify. APPROVE alone completes the issue; REJECT keeps the same lane and dispatches a fresh rework worker. A second REJECT with overlapping blockers stops the pack. Maker checks are evidence, never approval.

## Resume

The coordinator is disposable. On resume, the **root coordinator itself** sequences the existing source owners and owns the stop decision. First call the validated workspace parser/registry. In workspace mode, treat every issue repository name as a logical relative path: find its exact entry in the validated workspace profile, resolve `registered.path` against the validated workspace/artifact root to an absolute normalized path, then require that resolved absolute path to equal both its realpath and the canonical Git top-level **before resume Git checks**. Never compare the original relative string with an absolute realpath. Unknown names, symlinks, and non-root paths `STOP`. A canonical single repository skips logical-name binding and uses its already-absolute trusted root. Then use Implement's existing status/blocker scan to require exactly one blocker-resolved runnable issue; do not reproduce Loom's DAG or status vocabulary. Missing, duplicate, unknown, contradictory, or errored workspace or issue evidence is immediate `STOP` before dispatch.

For each dependency-ordered service lane, run Git only against that trusted canonical root with exact argument-array Git commands, never shell interpolation or issue-claimed paths. For every recorded Verify-approved commit/tree run `git cat-file -e <sha>^{commit}` and `git show -s --format=%T <sha>`; for each earlier approved commit run `git merge-base --is-ancestor <earlier> <lane-tip>`; require the latest approved lane tip to equal `git rev-parse HEAD` and its recorded tree to equal `git show -s --format=%T HEAD`; require `git status --porcelain` to be empty. Then use native story-filtered Orca inventory to require one exact active issue/lane/task identity consistent with the unique runnable issue, without duplicating Orca's task model. Any missing, duplicate, unknown, contradictory, dirty, command-error, or timeout result is `STOP` before worker or task dispatch. Give the complete decision a hard **180-second bound**. The root coordinator performs this comparison directly: never delegate any resume decision step to a helper, subagent, or worker. Only an explicit unique coordinator `CONTINUE` permits dispatch. Do not rely on coordinator memory or create a runtime/checkouts manifest, resume manifest, validator, or schema.

## Verified commit boundary

Before Verify, stage exactly intended files (including intended untracked content) without auto-staging unrelated files, then capture `git write-tree` and base SHA as the judged identity. After APPROVE only, create exactly one product-facing commit under the applicable explicit opt-in: the confirmed whole-pack preview is sufficient for that pack, while single-issue and other attended paths require their own explicit opt-in. Never push, publish, merge, amend, squash, rebase, or force. Compare `git rev-parse HEAD^{tree}` with the captured tree; mismatch requires re-Verify. If commit preceded reliable capture, Verify that exact commit SHA/tree. The digest records repository and service commit SHA.

Commit subject/body contain no private issue IDs, pack paths, model names, Loom branding, or orchestration mechanics. Explicit repository/project prose-language instructions win; otherwise use the current user's language. Git history informs style and conventions only, never language. One product-facing commit is created for each APPROVE, including each affected repository of an approved atomic multi-repository issue under that single issue boundary.

The stable lifecycle ends at a verified commit. In an attended invocation, only complete-pack Implement may then offer its separate exact Prepare review inventory and confirmation. A configured unattended invocation instead follows UNATTENDED setup/launch consent for its dedicated-branch and hosted-review exit; it does not enter attended Prepare review, and the attended confirmation does not apply to that invocation. Orca still owns branch naming, base, settings, attribution, repository/worktree IDs, and terminal lifecycle; Loom accepts those values and creates no publication manifest.

For a confirmed bundle, fetch and assess each lane's base without merge, rebase, or other history rewrite. Safe drift permits an ordinary hosted review; conflicts or a red current base block publication or limit that lane to a draft. Use `gh` only when it is available and the remote is GitHub; otherwise return an honest host-neutral manual outcome. Push and create at most one review per confirmed service, recording partial failures without rolling back successful reviews.

After the attempt, close exactly the confirmed completed maker/checker terminals, never the root coordinator before its publication summary. Approved Loom issues remain `Status: done`; `in-review` is only Orca native workspace state. Set `workspace-status: in-review` on exactly the listed worktree cards, and retain worktrees and branches until Tend proves merge and receives cleanup confirmation. Never auto-merge or auto-clean worktrees.

## Tend cleanup handoff

Explicit merged-worktree or local-lane cleanup routes to the existing Tend ritual. Tend first performs a read-only correlation of durable hosted-review merge evidence with exact Orca repository/worktree/card IDs, actual Git branch/HEAD and path, cleanliness, and active terminal/task/dispatch/rework state. A closed review, missing remote ref, issue status, or branch name is not merge proof. Missing, duplicate, orphaned, or contradictory disk/Git/Orca/host identity retains the lane for human resolution.

Only uniquely matched, durably merged, clean, inactive story-service lanes are removable by default, and only after a confirmation naming every local card/worktree/branch action. Tend removes that exact Orca-managed worktree only through native exact-selector `orca worktree rm`, then verifies the exact worktree/card is absent from both Orca and Git inventory. It retains closed-unmerged, dirty, blocked, active, rework, ambiguous, and orphaned lanes with an exact reason. If Orca cannot remove the worktree or either state cannot be verified, Tend retains the lane and branch and never falls back to `git worktree remove`, filesystem deletion, or another raw Git removal path. Only after verified native removal may Tend delete an exact still-present, still-eligible local branch with merged-safe deletion. It never deletes a remote branch by default, force-deletes a local branch, invokes `orca orchestration reset --all` or equivalent broad cleanup, or affects an unlisted terminal, task, worktree, or branch.

Cleanup is nontransactional per lane: preserve and report completed actions, identify partial failures, and rerun the exact read-only inventory before requesting confirmation for any retry. Internal consent chat may name exact local paths and IDs; product-facing summaries remain sanitized of private control-plane markers. Tend observes the human-performed merge and never crosses the human merge gate.
